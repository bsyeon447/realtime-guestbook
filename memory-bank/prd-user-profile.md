# 사용자 프로필 설정 기능 PRD

## 1. 개요

사용자 프로필 설정 기능은 로그인한 사용자가 자신을 나타내는 닉네임과 프로필 사진을 등록하고 관리할 수 있게 하는 기능이다. 이를 통해 방명록에서 누가 글을 작성했는지 한눈에 파악할 수 있고, 사용자 간 식별성과 기억 가능성을 높인다.

현재 방명록은 인증 사용자 기반으로 글 작성자를 연결하는 구조를 목표로 하고 있으므로, 프로필 기능은 `profiles` 데이터와 방명록 글 목록/상세 화면의 작성자 표시를 완성하는 역할을 한다.

## 2. 목표

* 로그인 사용자가 이메일 대신 원하는 닉네임으로 활동할 수 있게 한다.
* 사용자가 프로필 사진을 등록하여 방명록에서 시각적으로 식별되게 한다.
* 방명록 글 목록과 상세 화면에서 작성자 정보를 일관되게 표시한다.
* 회원가입 직후에도 기본 프로필 데이터가 자동 생성되어 프로필 참조 오류가 발생하지 않게 한다.

## 3. 사용자 스토리

**As a** 회원가입을 완료한 사용자로서,  
**I want to** 나를 표현하는 닉네임과 프로필 사진을 등록하고 싶다.  
**So that** 다른 사람들이 방명록에서 나를 쉽게 알아보고 기억할 수 있다.

## 4. 기능 요구사항

### 4.1 P0 필수 요구사항

#### P0-1. 프로필 페이지 진입

로그인한 사용자는 자신의 프로필 설정 페이지로 이동할 수 있어야 한다.

수용 기준:

* 로그인 상태에서 프로필 설정 페이지에 접근할 수 있다.
* 비로그인 사용자가 프로필 설정 페이지에 접근하면 로그인 페이지로 이동하거나 로그인 안내를 보여준다.
* 메인 방명록 화면에서 프로필 설정 페이지로 이동할 수 있는 진입점이 제공된다.

#### P0-2. 닉네임 변경

사용자는 이메일 아이디 대신 원하는 닉네임을 입력하고 저장할 수 있어야 한다.

수용 기준:

* 프로필 설정 페이지에서 현재 닉네임을 확인할 수 있다.
* 사용자는 닉네임을 수정하고 저장할 수 있다.
* 저장 후 방명록 화면에 변경된 닉네임이 반영된다.
* 닉네임이 비어 있는 경우 저장을 막거나 기본 표시명을 유지한다.

#### P0-3. 프로필 사진 업로드

사용자는 PC 또는 모바일에서 이미지를 선택해 프로필 사진으로 업로드할 수 있어야 한다.

수용 기준:

* 프로필 설정 페이지에서 이미지 파일을 선택할 수 있다.
* 업로드 성공 후 프로필 사진 미리보기가 갱신된다.
* 저장된 프로필 사진은 이후 재방문 시에도 유지된다.
* 이미지 업로드 실패 시 사용자가 이해할 수 있는 오류 메시지를 보여준다.

#### P0-4. 방명록 반영

방명록 글 목록과 상세 화면에서 작성자의 닉네임과 프로필 사진이 보여야 한다.

수용 기준:

* 방명록 글 카드에 작성자 닉네임이 표시된다.
* 프로필 사진이 있는 사용자는 사진이 함께 표시된다.
* 프로필 사진이 없는 사용자는 기본 아바타 또는 기본 표시 UI를 사용한다.
* 작성자가 기존 익명 데이터처럼 `user_id`가 없는 경우에도 화면이 깨지지 않는다.

#### P0-5. 프로필 자동 생성

회원가입 시 별도 설정 없이도 기본 프로필 데이터가 자동으로 생성되어야 한다.

수용 기준:

* 신규 사용자가 생성되면 `profiles` row가 자동 생성된다.
* 기본 `display_name`은 이메일 또는 이메일의 로컬 파트 등 식별 가능한 값으로 설정된다.
* 프로필 자동 생성 실패가 글 작성 흐름의 장애로 이어지지 않도록 확인한다.

## 5. P1 권장 요구사항

### P1-1. 닉네임 중복 방지

다른 사용자와 동일한 닉네임은 사용할 수 없게 제한한다.

수용 기준:

* 이미 사용 중인 닉네임 저장을 시도하면 저장이 실패한다.
* 중복 오류는 사용자에게 명확하게 표시된다.
* 대소문자 또는 공백 처리 기준을 정하고 일관되게 적용한다.

### P1-2. 이미지 용량 제한

너무 큰 프로필 사진은 업로드되지 않도록 제한한다.

수용 기준:

* 5MB 이상의 이미지는 업로드 전에 클라이언트에서 차단한다.
* Storage 정책 또는 업로드 로직에서도 용량 제한을 보조적으로 검증한다.
* 제한 초과 시 사용자가 파일 크기 문제를 이해할 수 있는 오류 메시지를 보여준다.

## 6. 데이터 모델 요구사항

프로필 정보는 `public.profiles` 테이블을 기준으로 관리한다.

필수 필드:

* `id`: `auth.users.id`와 연결되는 사용자 UUID
* `display_name`: 방명록에 표시할 닉네임
* `avatar_url`: 프로필 사진 경로 또는 public URL
* `updated_at`: 마지막 수정 시각

권장 제약:

* `display_name`은 빈 문자열을 허용하지 않는다.
* P1 구현 시 `display_name`에 unique 제약 또는 정규화된 닉네임 컬럼을 추가한다.
* 프로필 사진은 Supabase Storage의 사용자별 경로에 저장한다.

## 7. 권한 및 보안 요구사항

* 사용자는 본인 프로필만 수정할 수 있어야 한다.
* 모든 사용자는 방명록 표시를 위해 공개 프로필 정보(`display_name`, `avatar_url`)를 조회할 수 있어야 한다.
* 다른 사용자의 프로필 수정은 RLS 정책으로 차단해야 한다.
* 프로필 사진 업로드 경로는 사용자 ID를 포함해 사용자별로 분리한다.
* 서비스 롤 키는 브라우저에서 사용하지 않는다.

## 8. 화면 및 UX 요구사항

프로필 설정 페이지는 최소한 다음 요소를 포함한다.

* 현재 프로필 사진 또는 기본 아바타
* 프로필 사진 업로드 입력
* 닉네임 입력 필드
* 저장 버튼
* 저장/업로드 진행 상태
* 성공 및 실패 메시지

방명록 화면은 최소한 다음 요소를 포함한다.

* 작성자 닉네임
* 작성자 프로필 사진 또는 기본 아바타
* 기존 익명 글에 대한 fallback 표시

## 9. 비범위

이번 PRD의 1차 구현 범위에는 다음을 포함하지 않는다.

* 팔로우/친구 기능
* 공개 프로필 페이지
* 프로필 소개글, 링크, SNS 계정
* 이미지 크롭/편집 기능
* 관리자용 프로필 관리 UI

## 10. 구현 메모

* 현재 인증/인가 계획의 `profiles` 테이블과 연계해 구현한다.
* 방명록 조회 시 `posts`와 `profiles`를 함께 조회하거나, 작성자 정보 표시용 뷰를 검토한다.
* 기존 `PostItGrid`는 작성자 표시 영역을 확장해야 한다.
* 프로필 사진 Storage 버킷은 기존 `guestbook` 버킷을 재사용할지, `avatars` 전용 버킷을 만들지 결정해야 한다.
* 전용 버킷을 만들 경우 RLS/Storage 정책을 별도로 작성한다.

## 11. 완료 기준

* 로그인 사용자가 프로필 설정 페이지에서 닉네임과 사진을 저장할 수 있다.
* 신규 가입 사용자의 기본 프로필이 자동 생성된다.
* 방명록 글 목록과 상세 화면에 작성자의 닉네임과 프로필 사진이 표시된다.
* 비로그인 사용자는 프로필 설정을 수정할 수 없다.
* 사용자는 본인 프로필만 수정할 수 있다.
* P1 범위 구현 시 닉네임 중복과 5MB 이상 이미지 업로드가 차단된다.
